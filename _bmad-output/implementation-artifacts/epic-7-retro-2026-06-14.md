## Story Analysis

### Story 7.1 — Operator Metrics Dashboard

**What happened:** Clean foundational story. Established the admin infrastructure pattern: `role` column on `profiles`, service-role client, `getAdminMetrics()` shared function, admin layout role guard, and the `get_dau` Postgres RPC (avoiding PostgREST's 1k row limit). No code review findings — the team applied all prior patterns correctly on the first attempt.

**Highlight:** Using a Postgres RPC for DAU count was the right architectural call. PostgREST silently caps rows at 1,000 — without the RPC, DAU would have been capped at 1,000 distinct users with no error.

### Story 7.2 — Per-User Email Delivery Lookup

**What happened:** Six code review patches applied, two high-severity.

- **P1 (High) — Silent error swallowing:** Only `briefingsResult.error` was checked; errors from `reengagements` and `profiles` queries were silently ignored. A production DB failure would return partial data as if it were complete.
- **P2 (High) — Service role grant SELECT-only:** The `reengagement_notifications` migration granted only SELECT to `service_role`. Insert/update/delete from the Inngest functions would have failed silently in production.
- **P3 (Med) — Audit log missing on UI path:** `getAdminUserData()` was only called from the API route; the RSC page called it without passing `adminUserId`, so UI-initiated lookups produced no audit row.
- **P4 (Med) — Auth error conflated with not-found:** A Supabase auth service failure returned 404 NOT_FOUND instead of 502 AUTH_ERROR, hiding operational failures.
- **P5 (Med) — Validation after role DB query:** UUID validation ran after the `profiles.role` DB query — invalid UUIDs hit the DB unnecessarily. Fix: validate input before any DB call.
- **P6 (Low) — Hydration mismatch:** `toLocaleString()` without locale/timezone arguments produces different output in Node vs browser, causing React hydration errors.

**Standing lesson from P5:** "Validate input before the role DB query." This became a codified pattern applied correctly in 7.3 on first attempt.

### Story 7.3 — System-Wide Broadcast

**What happened:** Most complex story in the epic. Two rounds of adversarial code review; 10 patches applied across both rounds; 5 items deferred as pre-existing or acceptable at current scale.

**Round 1 (8 patches):**

- **HTML injection:** Admin-authored subject and body were interpolated directly into email HTML without escaping. The `escapeHtml` function existed in the codebase (`lib/email/templates/dataExport.ts`) but was not applied. Fixed in `lib/email/templates/broadcast.ts`.
- **Fan-out scale:** `find-recipients` queried without pagination (Supabase silently caps at 1,000 rows); `Promise.all` spawned one `step.run` per user with no batching (Inngest ~1,000-step ceiling). Fixed with `PAGE_SIZE=1000` paginated loop + `BATCH_SIZE=100` batch fan-out.
- **Complete profiles filter:** AC3 required sending only to users with complete profiles. `find-recipients` used goals-table membership as the proxy for profile completeness, which satisfies the spec given the onboarding flow. Implemented.
- **Non-durable audit log:** The `admin_broadcast_sent` audit insert was fire-and-forget outside any `step.run`. On Inngest retry (with `retries: 3`), the insert would run multiple times. Fixed by wrapping in `step.run("write-audit-log")`.
- **Audit log errors swallowed:** Matched the established `.then(({ error }) => …).catch(…)` logging pattern from `app/api/admin/users/route.ts`.
- **Skip recipient logging:** No structured log when a recipient is skipped (no email / unconfirmed). Added `broadcast_recipient_skipped` event, matching `checkInactivity`'s `NO_EMAIL` log pattern.
- **Whitespace-only body:** `min(1)` counts whitespace as content. Added `.trim()` to `AdminBroadcastSchema.body`.
- **401 message inconsistency:** Used `"Authentication required."` instead of `"Not authenticated"` — fixed to match every other route.

**Round 2 (2 patches — caught by patch-verification review):**

- **Unsubscribe href unescaped:** Round 1 applied `escapeHtml` to body paragraphs and COMPANY_ADDRESS but missed the `unsubscribeUrl` in the `href` attribute. The URL `?token=X&userId=Y&type=Z` has unescaped `&` — strict HTML parsers (including many email clients) strip `userId` and `type` params, returning 400 on every unsubscribe click. This would have silently broken CAN-SPAM compliance. Fixed: `href="${escapeHtml(unsubscribeUrl)}"`.
- **Subject missing `.trim()`:** Round 1 added `.trim().min(1)` to `body` but not `subject`. A whitespace-only subject like `"   "` passed `min(1)`. Fixed.

**Deferred (5 items):**

| Item | Rationale |
|---|---|
| CSRF / same-origin enforcement on mutating admin POSTs | Pre-existing project-wide gap |
| Rate limiting on admin endpoints | Pre-existing project-wide gap |
| Role check before validation (spec says "admin role verified first") | Pre-existing ordering inconsistency; non-admin gets VALIDATION_ERROR instead of FORBIDDEN |

| Inngest step ceiling at N=99,801 recipients (1+⌈N/100⌉+1=1000) | Theoretical at current scale |

## What Needs Improvement

### 1. HTML security pass not systematic
`escapeHtml` existed in the codebase but wasn't applied to the broadcast template until review caught it. The unescaped href survived Round 1 and required Round 2. Every task that builds HTML templates must include an explicit step: "escape all interpolated values with `escapeHtml`."

**Action:** Add to the story template's dev notes checklist: "HTML templates: verify every interpolated value uses `escapeHtml` before merging."

### 2. DB error completeness not systematic
Silent error swallowing appeared in both 7.2 (only one of three DB errors checked) and 7.3 (audit log error swallowed). This is a recurring review finding.

**Action:** Add to the story template's dev notes checklist: "Every `.from().select/insert/update/delete` call must check `.error` and either throw, log, or handle explicitly."

### 3. Scale considerations not asked at story creation time
The 7.3 fan-out issue (unbounded query, no batching) was foreseeable at story-writing time. The question "what's the upper bound of this query?" was not asked until the code review.

**Action:** Story creation prompt should ask: "Does this story include any query or fan-out over a user set? If so, document the expected upper bound and pagination/batching strategy in Dev Notes."

### 4. Deferred security items accumulating
CSRF enforcement, rate limiting, and role-check ordering were all deferred as "pre-existing." These are now documented in `deferred-work.md` but have no owner.

**Action:** A security hardening story should be added to the Phase 2 backlog before user testing begins.
